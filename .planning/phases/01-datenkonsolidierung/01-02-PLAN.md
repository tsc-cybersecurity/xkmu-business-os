Das Script hat folgende Eigenschaften:
- Laeuft in einer einzigen BEGIN/COMMIT Transaktion
- Idempotenz-Guard am Anfang: prueft ob default-Tenant noch existiert
- SAVEPOINT fuer jede Tabellen-Gruppe (tracebar bei Fehler)

- Validierungs-Queries am Ende (keine DML, nur SELECTs die Probleme aufdecken)

-- ============================================================
-- Migration 002: Tenant-Konsolidierung
-- Default → xkmu-digital-solutions Merge
--
-- Source (loeschen): default        2ce4949e-8017-4d26-9d60-66c3f4060673
-- Ziel   (behalten): xkmu-digital   7b6c13c5-1800-47b2-a12f-10ccb11f6358
--
-- Idempotent: Prueft ob Default-Tenant noch existiert.
-- Bei erneutem Ausfuehren: kein Fehler, kein Datenverlust.
--
-- Backup: Erfolgt durch entrypoint.prod.sh (pg_dump) BEVOR diese Migration laeuft.
-- ============================================================

BEGIN;

-- ============================================================
-- IDEMPOTENZ-GUARD
-- Wenn default-Tenant nicht mehr existiert: alles ueberspringen.
-- ============================================================
DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM tenants WHERE id = '2ce4949e-8017-4d26-9d60-66c3f4060673'
  ) THEN
    RAISE NOTICE '002_tenant_consolidation: Default-Tenant existiert nicht mehr — Migration bereits ausgefuehrt. Ueberspringe.';
    -- Wir koennen hier nicht einfach "RETURN" aus dem Transaktionsblock.
    -- Stattdessen setzen wir eine Session-Variable.
    PERFORM set_config('app.migration_002_skip', 'true', true);
  ELSE
    PERFORM set_config('app.migration_002_skip', 'false', true);
    RAISE NOTICE '002_tenant_consolidation: Default-Tenant gefunden. Starte Migration.';
  END IF;
END $$;

-- ============================================================
-- SAVEPOINT SP_ROLES
-- Strategie: DEDUP_THEN_MERGE
-- Duplikate: owner, admin, member existieren in beiden Tenants.
-- xkmu-Rollen bleiben. Default-Rollen werden geloescht.
-- role_permissions: CASCADE via role_id automatisch mitgeloescht.
-- users.roleId: ON DELETE SET NULL → wird NULL gesetzt.
-- ============================================================
DO $$
BEGIN
  IF current_setting('app.migration_002_skip', true) = 'true' THEN RETURN; END IF;

  SAVEPOINT sp_roles;

  RAISE NOTICE 'SP_ROLES: Loeschen von Duplikat-Rollen in default-Tenant...';

  -- Loeschen der default-Rollen die gleichnamig in xkmu existieren
  DELETE FROM roles
  WHERE tenant_id = '2ce4949e-8017-4d26-9d60-66c3f4060673'
    AND name IN (
      SELECT name FROM roles
      WHERE tenant_id = '7b6c13c5-1800-47b2-a12f-10ccb11f6358'
    );

  -- Verbleibende Rollen im default-Tenant zu xkmu migrieren
  UPDATE roles
  SET tenant_id = '7b6c13c5-1800-47b2-a12f-10ccb11f6358'
  WHERE tenant_id = '2ce4949e-8017-4d26-9d60-66c3f4060673';

  RAISE NOTICE 'SP_ROLES: Fertig.';
END $$;

### Teil B: docker/app/entrypoint.prod.sh

Die Migration muss auch im Pre-Drizzle SQL-Block laufen (Zeile 98, innerhalb des EOSQL-Heredocs).
Das stellt sicher dass sie VOR dem Drizzle-Schema-Sync laeuft.

ACHTUNG: Den EOSQL-Block NICHT ersetzen — nur einen neuen Abschnitt VOR der letzten EOSQL-Zeile
(also VOR "EOSQL") einfuegen.

Suche die Zeile `EOSQL` am Ende des Pre-Drizzle-Blocks (ca. Zeile 273 nach den execution_logs-Indizes).
Fuge direkt DAVOR ein:

```sql
-- ── 002: Tenant-Konsolidierung (idempotent — prueft ob default-Tenant noch existiert) ──
-- Diese Migration ist sicher mehrfach ausfuehrbar.
-- Wenn default-Tenant nicht existiert: komplett no-op.
DO $$
DECLARE
  default_tenant_id UUID := '2ce4949e-8017-4d26-9d60-66c3f4060673';
  xkmu_tenant_id UUID := '7b6c13c5-1800-47b2-a12f-10ccb11f6358';
BEGIN
  IF NOT EXISTS (SELECT 1 FROM tenants WHERE id = default_tenant_id) THEN
    RAISE NOTICE '002_tenant_consolidation (entrypoint): Default-Tenant nicht gefunden — Migration bereits ausgefuehrt. Ueberspringe.';
    RETURN;
  END IF;
  -- Die Migration wird durch src/lib/db/migrations/index.ts registriert und
  -- durch instrumentation.ts/execute-plan ausgefuehrt (002_tenant_consolidation.sql).
  -- Dieser Block hier ist nur ein Idempotenz-Hinweis fuer Ops.
  RAISE NOTICE '002_tenant_consolidation (entrypoint): Default-Tenant gefunden — Migration wird via _migrations-System ausgefuehrt.';
END $$;
```

HINWEIS fuer den Executor: Der entrypoint-Block oben ist ein reiner Hinweis-Block.
Die eigentliche Migration laeuft ueber das _migrations-System (instrumentation.ts liest
MIGRATIONS aus index.ts und fuehrt die SQL-Dateien aus). Der Pre-Drizzle-Block im
entrypoint ist NICHT der richtige Ort fuer die volle Migration — sie ist zu lang und
komplex fuer ein Heredoc. Der entrypoint-Block stellt nur sicher dass ein Op der
direkt psql aufruft einen sinnvollen NOTICE-Text sieht.
