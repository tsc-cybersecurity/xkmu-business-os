### OrderCategoryService (simpel)

```ts
export const OrderCategoryService = {
  async list(activeOnly = true): Promise<OrderCategory[]> {
    const conditions = activeOnly ? [eq(orderCategories.isActive, true)] : []
    return db.select().from(orderCategories)
      .where(conditions.length ? and(...conditions) : undefined)
      .orderBy(asc(orderCategories.sortOrder), asc(orderCategories.name))
  },
  async getById(id: string): Promise<OrderCategory | null> {
    const [row] = await db.select().from(orderCategories).where(eq(orderCategories.id, id)).limit(1)
    return row ?? null
  },
}
```

### OrderService

Methoden:
- `create(input: { companyId, requestedBy, categoryId?, title, description, priority, contractId?, projectId? }): Promise<Order>`
- `list(filter: { companyId?; status?; priority?; categoryId?; assignedTo?; limit?; offset? }): Promise<Order[]>`
- `getById(id: string): Promise<Order | null>`
- `cancel(id: string, requestedBy: string): Promise<boolean>` — nur eigene pending
- `transitionStatus(id: string, action: 'accept'|'start'|'complete'|'reject', rejectReason?: string): Promise<Order>` — validiert erlaubte Übergänge
- `assign(id: string, assignedTo: string | null): Promise<Order>`

Status-Transition-Rules (als lokale Map):
```ts
const ALLOWED_TRANSITIONS: Record<string, string[]> = {
  pending:     ['accepted', 'rejected'],
  accepted:    ['in_progress', 'rejected'],
  in_progress: ['done', 'rejected'],
  done:        [],
  rejected:    [],
  cancelled:   [],
}
```

Action → new Status Mapping:
- `accept` → `accepted` (setzt `acceptedAt`)
- `start` → `in_progress` (setzt `startedAt`)
- `complete` → `done` (setzt `completedAt`)
- `reject` → `rejected` (setzt `rejectedAt` + `rejectReason`)

Validation: if current status is not in ALLOWED_TRANSITIONS for the target, throw `Error('INVALID_TRANSITION')`.

`cancel` setzt `status='cancelled'` + `cancelledAt`, DB-Constraint: only `requestedBy` AND `status='pending'` (im WHERE-Clause).

### Tests (min 10)

- create inserts with defaults
- list filters by companyId, status, priority, categoryId
- getById returns row / null
- cancel returns true on own pending
- cancel returns false when wrong owner or wrong status
- transitionStatus happy path (accept, start, complete, reject)
- transitionStatus throws INVALID_TRANSITION on invalid jump (e.g., pending→done)
- transitionStatus sets correct timestamp
- reject without reason → still allowed (reason optional at service, enforced at API layer as min-1-char)
- assign sets assignedTo, returns updated

Use `setupDbMock` pattern from `user-service-portal.test.ts`.

## Task P4-3: Portal-APIs (create/list/detail/cancel + Audit)

**Files:**
- Create: `src/app/api/v1/portal/me/orders/route.ts` (GET list, POST create)
- Create: `src/app/api/v1/portal/me/orders/[id]/route.ts` (GET detail, DELETE cancel)

### Zod Create-Schema

```ts
const createSchema = z.object({
  categoryId: z.string().uuid().optional(),
  title: z.string().min(3).max(255),
  description: z.string().min(10),
  priority: z.enum(['hoch', 'mittel', 'niedrig', 'kritisch']).default('mittel'),
  contractId: z.string().uuid().optional(),
  projectId: z.string().uuid().optional(),
}).strict()
```

### POST create

Nach Validation:
1. Wenn `contractId` gesetzt: query that `documents.id = contractId AND companies.id = auth.companyId AND type='contract'` — sonst 400 "VALIDATION_ERROR, Vertrag gehört nicht zu Ihrer Firma".
2. Wenn `projectId` gesetzt: analog gegen `projects`.
3. Wenn `categoryId` gesetzt: prüfe `orderCategories.id` existiert und `isActive=true` — sonst 400.
4. `OrderService.create({ ...data, companyId: auth.companyId, requestedBy: auth.userId })`.
5. `AuditLogService.log({ action: 'portal.order_created', entityType: 'order', entityId: order.id, payload: { title, categoryId, priority } })` (fail-safe).
6. Queue `portal_order_created_admin` email to org-email.
7. Return 201 with masked order (no internal fields).

### GET list
`OrderService.list({ companyId: auth.companyId, limit: 100 })` + join mit `orderCategories.name` um das Kategorien-Label gleich mitzuliefern. Alternativ: client macht zweite Query. Für P4: join-Variante direkt im Route-Handler (einfacher select with join).

### GET detail
Select mit joins zu `orderCategories`, `documents` (für Vertrags-Nummer), `projects` (für Name). Nur Felder die dem Portal was nützen — `assignedTo`, interne Timestamps bleiben drin (Kunde sieht Lebenszyklus-Daten).

### DELETE cancel
`OrderService.cancel(id, auth.userId)`. Wenn false → 404. Bei Erfolg: audit `portal.order_cancelled`.

### Portal order-categories list
`OrderCategoryService.list(true)` mit Projektion `{id, name, slug, color}`.

**Files:**
- Create: `src/app/api/v1/orders/route.ts` (GET global list)
- Create: `src/app/api/v1/orders/[id]/route.ts` (GET detail, PATCH status/assign)

### GET detail (`users:read`)
Select mit vollen Kontext-Joins (Kategorie, Firma, User, Vertrag, Projekt).

### PATCH (`users:update`)
Discriminated-Union zod body:

```ts
const patchSchema = z.discriminatedUnion('action', [
  z.object({ action: z.literal('accept') }),
  z.object({ action: z.literal('start') }),
  z.object({ action: z.literal('complete') }),
  z.object({ action: z.literal('reject'), rejectReason: z.string().min(1).max(1000) }),
  z.object({ action: z.literal('assign'), assignedTo: z.string().uuid().nullable() }),
])
```

Je nach action → OrderService.transitionStatus oder .assign. Audit-Log für jede Änderung. Bei transition: queue `portal_order_status_changed` email an requestedBy. Bei assign: kein kundeseitiges Mail (intern).
